# Тестовое задание: корректность приведенного примера 
При сравнении своих результатов с приведенным в файле корректным результатом обнаружил несоответсвия, причиной оказалась в трех, возможно, ошибках, или, что тоже врозможно, я сам не совсем корректно разобрался с условием задачи.

## 1. Входные данные: комнаты 3 и 4

По заданию комнаты представляют собой неориентированный граф. Тогда если вершина A связана c B, то и вершина B должна быть связана с A. Данное усливе не выолняется для вершин 3 - 4. А в выходных данных бот идет из 3-ей комнаты в 4-ую, что при таких данных невозможно. Бот застревает в комнате 4, так как назад он может идти только по посещенным вершинам, из всех соседних посещенная вершина только 3, а по входным данным из 3 в 4 дороги нет. Бот ведет себя подобным образом как в выходных данных, если добавить к соседям 4-ой комнаты 3-ю комнату.

## 2. Выходные данные: итоговая сумма

По условию задачи, ценности ресурсов имеют следующие значения:
```text
         iron  gold  gems  Exp
Ценность  7     11    23    1
```


```text
Проанализируем выходные данные:
Целевой ресурс - gems - то есть цена его в 2 раза больше
Собранные ресурсы:
    в 1 комнате 1 gems
    в 3 комнате 2 gems
    в 4 комнате 4 gold
```

Считаем сумму: 1 * 23 * 2 + 2 * 23 * 2 + 4 * 11 = 182.
А в выходны данных общая ценность равна 93

# Улучшения алгоритма бота Алисы

## 1. Выбор пути возврата в 0 комнату с учётом смежных комнат

**Текущая реализация:** бот возвращается только по посещённым комнатам кратчайшим путём.

**Предлагаемое улучшение:** при построении маршрута возврата анализировать не только посещённые, но и видимые комнаты. Если через видимую комнату путь до старта короче, то пройти через неё.

**Положительные стороны:**
- Можно найти более короткий путь до старта, сэкономив еду
- Можно собрать ресурсы в ранее не посещённых, но видимых комнатах (причем это даже не затратит еды, так как в первый раз будем собирать еду)

**Отрицательные стороны:**
- Риск: ресурсы могут быть менее ценными чем в уже известных комнатах. То есть может быть ситация, что лучше потратить еду на сбор бОльшего количетсва ресурсов, чем бесплатно собирать по чуть-чуть.

**Сценарии:**
- Помогает: когда видимая комната сокращает путь (когда мы знаем что гарантировано она скоратит путь)
- Не помогает: когда видимая комната ведёт в неизвестную область без выхода



## 2. Пропуск сбора ресурсов в пустых комнатах

**Текущая реализация:** если в комнате нет ресурсов, бот всё равно пытается выполнить сбор. Причем если это стадия возрата, то мы тратим лишнюю еду (если она есть) на сбор - то есть бот в пустой комнате может потратить до 3 единиц еды на сбор "пустых" ресурсов

**Предлагаемое улучшение:** явно проверять наличие ресурсов перед сбором. Если ресурсов нет - немедленно переходить к следующему действию (движение)

**Реализовано:** если в комнате нет ресурсов, то сразу помечается, что все ресурсы в данной комнате собраны

**Положительные стороны:**
- Экономия еды (нет лишнего шага)
- Ускоряет работу бота

**Сценарии:**
- Помогает: в подземельях с большим количеством пустых комнат
- Не помогает: когда все комнаты содержат ресурсы



## 3. Завершение исследования при посещении всех известных комнат

**Текущая реализация:** бот переходит в фазу возврата только по условию "потрачена половина еды".

**Предлагаемое улучшение:** добавить второе условие - если все видимые комнаты посещены, переходить к возврату независимо от оставшейся еды.

**Реализовано:** бот проверяет, если путь до ближайщей непосещенной комнаты пуст (то есть все известные комнаты посещены), то фаза меняется на стадию возврата.

**Положительные стороны:**
- Не тратит еду на бесцельное хождение (возможно непредвиденное поведение, что бот будет делать в такой ситуации?)
- Позволяет сохранить больше еды для сбора излишков на обратном пути

**Сценарии:**
- Помогает: в маленьких подземельях или при много бОльшем количестве еды в мАлом подземелье. Когда комнаты максимально соеденины друг с другом (то есть дорога до ближайщей непосещенной будет достаточно мала и еды на это много не требуется)
- Не помогает: когда есть длинные цепочки комнат


## 4. Защита от зацикливания при возврате через стартовую комнату и алализ рациаональности пути относительно сбора

**Проблема:**   3
                0 1,2 0 0 0 0
                1 0,3 5 2 1 15
                2 0 0 3 0 0
                3 1 2 0 0 0
                8 iron
При таких данных бот Алисы пройдет 1 и 3 комнаты, будет больше половины еды. Следующая комната из ближайших непосещенных будет 2, пока он будет идти к комнате 2, еда станет равна половине в комнате 0. А по алгоритму Алисы вся еда оставшаяся (половина) должна потратится на то, чтобы попасть в 0 комнату (в которой  бот уже нахродится).

**Предлагаемое улучшение:** Добавлять анализ для каждой дороги в новую непосещенную вершину при наличии в дороге комнаты 0. А также делать приоритет на сбор ресурсов в уже ивзестных комнатах при достаточно длинном пути относительно текущего количества еды (добавить анализ на то, что будет рациональнее, собирать в ближайщих посещенных вершинах еду или все таки идти до дальней непосещенной комнаты)

**Положительные стороны:**
- Экономия еды
- Можно собрать в итоге больше ресурсов

**Отрицательные стороны:**
- В дальней комнате может очень большое скопление ресурсов, в таком случае стоило бы идти до нее. Но такое будет встречаться реже.



## 5. Оптимизация сбора излишков еды на обратном пути

**Текущая реализация:** бот собирает ресурсы в порядке убывания ценности, но не учитывает данные о посещенных комнатах, через которые будет идти до начальной (нулевой) комнаты. 

**Предлагаемое улучшение:** при переходе в стадию возврата строится путь по посещенным комнатам - то есть боту известны ресурсы в каждой комнате по дороге. Можно добавить анализ этих данных, чтобы если была излишняя еда, то ресурсы брались не с текущей комнаты, а вместо этого строился отсортированный список где хранилась ценность/ресурс/комната и относительно этого рассчитывалось где лучше потратить еду на сбор ресурсов

**Положительные стороны:**
- Более эффективный расход еды
- Собирает больше ценных ресурсов на обратном пути

**Отрицательные стороны:**
- Усложнение алгоритма и увеличение времени работы

**Сценарии:**
- Помогает: когда ценные ресурсы разбросаны по подземелью
- Не помогает: когда все ценные ресурсы сконцентрированы в одной комнате



## 6. Проверка симметричности графа при загрузке

**Текущая реализация:** граф загружается как есть, без проверки симметричности рёбер.

**Предлагаемое улучшение:** при загрузке проверять: если комната A имеет соседа B, то комната B должна иметь соседа A. Если данное условие не выполняется, то либо автоматически добавлять обратную связь, либо сообщать об ошибке.

**Положительные стороны:**
- Предотвращает ошибки в поиске пути через непосещённые комнаты и в целом в ходен всего алгоритма

**Отрицательные стороны:**
- Дополнительное время загрузки

**Сценарии:**
- Помогает: когда входные данные содержат ошибки или несимметричны
- Не помогает: когда гарантируется, что граф ориентированный
